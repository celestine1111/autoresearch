> **CODE WORD: When the user starts a prompt with `/seobetter` — READ ALL 5 .md files before coding.** (plugin_UX, article_design, plugin_functionality_wordpress, SEO-GEO-AI-GUIDELINES, external-links-policy)
>
> **MANDATORY PRE-CHECK: Before making ANY changes to Content_Formatter.php, Stock_Image_Inserter.php, Citation_Pool.php, seobetter.php, or Async_Generator.php, READ THIS FILE, plugin_UX.md, AND external-links-policy.md first. Verify no features are removed after changes.**
>
> **Purpose:** Defines how generated articles look, render, and perform across any CMS. Every article generated by SEOBetter follows this specification exactly.
>
> **Referenced by:** `Content_Formatter.php` (classic + hybrid modes), `Stock_Image_Inserter.php` (image placement), `Citation_Pool.php` (references), `Async_Generator.php`, `seobetter.php`
>
> **CRITICAL:** Saved drafts use `format_hybrid()` — native Gutenberg blocks for editable content (headings, paragraphs, lists, images) + `wp:html` blocks with inline styles for styled elements (takeaways, pros/cons, tables, callouts, blockquotes). If styled elements have no styling, check that `format_hybrid()` is detecting context from preceding headings. If the entire article is uneditable, check that `rest_save_draft()` calls `format('hybrid')` not `format('classic')`.
>
> **CRITICAL — BANG-BUG (FM-13):** All regex patterns that match markdown links `[text](url)` in `seobetter.php`, `Content_Formatter.php`, and `AI_Content_Generator.php` MUST use the `(?<!!)` negative lookbehind to avoid matching the inner `[alt](url)` of image markdown `![alt](url)`. Breaking this invariant causes stripped images with stray `!` characters, which cascade into broken Key Takeaways styling. See `external-links-policy.md` FM-13.

## 1. DESIGN PHILOSOPHY

Articles must look like they came from a design-conscious publication (The Verge, Stripe Press, NYT Magazine) — not a Notion page full of emoji headers or a chatbot response.

### Core Principles
1. **Hybrid format** — Native Gutenberg blocks for editable content + `wp:html` blocks with inline styles for styled elements only.
2. **Editable** — Headings, paragraphs, standard lists, images are native WordPress blocks. Users can edit them in the block editor.
3. **Styled where needed** — Key takeaways, pros/cons, tables, callouts, blockquotes use `wp:html` with inline styles and `!important` to override theme CSS.
4. **Schema injected** — JSON-LD schema is appended as a `wp:html` block at the end of every article's `post_content`.
5. **Responsive** — Responsive images with `loading="lazy"`, mobile-friendly tables with `overflow-x:auto`.
6. **Accessible** — Semantic HTML, sufficient contrast ratios, no color-only indicators.
7. **Theme-compatible** — Uses `!important` on styled elements to override any WordPress theme CSS. Editable blocks inherit theme fonts and colors.

## 2. HTML STRUCTURE (Hybrid Format)

### Saved Draft Structure
```
<!-- wp:heading -->                    ← EDITABLE
<h1 class="wp-block-heading">Title</h1>
<!-- /wp:heading -->

<!-- wp:paragraph {"fontSize":"small"} -->   ← EDITABLE
<p class="has-small-font-size"><em>Last Updated: April 2026</em></p>
<!-- /wp:paragraph -->

<!-- wp:heading -->                    ← EDITABLE
<h2 class="wp-block-heading">Key Takeaways</h2>
<!-- /wp:heading -->

<!-- wp:html -->                       ← STYLED (inline CSS)
<div style="border-left:4px solid #764ba2;background:linear-gradient(...)">
  <ul><li>...</li></ul>
</div>
<!-- /wp:html -->

<!-- wp:heading -->                    ← EDITABLE
<h2 class="wp-block-heading">Section</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->                  ← EDITABLE
<p>Content text...</p>
<!-- /wp:paragraph -->

<!-- wp:list -->                       ← EDITABLE
<ul><li>Item</li></ul>
<!-- /wp:list -->

<!-- wp:html -->                       ← STYLED TABLE
<div style="overflow-x:auto"><table>...</table></div>
<!-- /wp:html -->

<!-- wp:html -->                       ← SCHEMA
<script type="application/ld+json">{"@context":"https://schema.org",...}</script>
<!-- /wp:html -->
```

### Preview Structure (Classic Format)
The plugin preview uses `format_classic()` which outputs a self-contained `<div>` with scoped `<style>` block for the admin preview panel only. This is NOT what gets saved to WordPress.

```html
<style>
  .sb-[uid] { /* all article styles scoped here */ }
  .sb-[uid] h1, .sb-[uid] h2 { /* heading styles */ }
  /* ... all styles prefixed with .sb-[uid] ... */
  @media (prefers-color-scheme: dark) {
    .sb-[uid] { /* dark mode overrides */ }
  }
</style>
<article class="sb-[uid]">
  <p style="...">Last Updated: Month Year</p>
  <h1>Article Title</h1>
  <h2>Key Takeaways</h2>
  <div class="sb-takeaways">
    <ul><li>...</li></ul>
  </div>
  <!-- content sections -->
  <h2>Section Heading</h2>
  <p>Content paragraph...</p>
  <!-- ... -->
  <h2>References</h2>
  <ol><li><a href="https://real-url.com">Source Title</a> — description</li></ol>
</article>
```

### Rules
- Output a single `<article>` element with class `sb-[6-char-uid]`
- Include one `<style>` tag at the top with every selector prefixed by the scoping class
- Do NOT use Tailwind, Bootstrap, or any framework classes
- Do NOT use `<link>` tags, `@import`, or `<script>` tags
- Return only the HTML block — no markdown fences, no DOCTYPE

## 3. TYPOGRAPHY

### Font Stacks (System Fonts — No External Fonts)
| Context | Font Stack |
|---|---|
| Headings | `ui-serif, Georgia, 'Times New Roman', serif` |
| Body text | `ui-sans-serif, system-ui, -apple-system, sans-serif` |
| Code | `ui-monospace, 'Cascadia Code', 'Fira Code', monospace` |

### Font Sizes (Fluid via clamp)
| Element | Size | Weight | Line Height |
|---|---|---|---|
| H1 | `clamp(1.8em, 4vw, 2.4em)` | 800 | 1.2 |
| H2 | `clamp(1.3em, 3vw, 1.6em)` | 700 | 1.3 |
| H3 | `1.15em` | 600 | 1.4 |
| Body paragraph | `1.05em` | 400 | 1.75 |
| Table header | `0.9em` | 600 | — |
| Table cell | `0.95em` | 400 | — |
| Small/muted | `0.85em` | 400 | — |
| Code block | `0.9em` | 400 | 1.6 |

### Text Wrapping
- Headings: `text-wrap: balance`
- Paragraphs: `text-wrap: pretty`
- Body max-width: `65ch` (optimal reading length)

### Drop Cap
The opening paragraph of the article gets a `::first-letter` drop cap:
- Font size: `3.2em`
- Color: accent color
- Font family: serif (heading stack)
- Float left with small right margin

- Inline code: light gray background, small padding, rounded corners
- Code blocks: dark background (#1f2937), light text, rounded corners, horizontal scroll

## 6. ICON RULES (STRICT)

### Inside Generated Articles
- **NEVER** place icons next to headings, list items, or inline with sentences
- **NEVER** use emoji as icons in body copy
- No checkmark bullets, no decorative icons beside H2s, no emoji headers
- This is the single clearest tell that content was AI-generated

### Icons Permitted ONLY In
1. Small SVG glyphs inside callout box corners (info icon, sized 1em, `currentColor`)
2. A single icon in the Key Takeaways box header (not per item)
3. Social share or author byline sections

### When Icons Are Used
- Embed as inline SVG directly in the HTML
- Do NOT reference Lucide or Iconify as runtime dependencies
- Copy raw SVG path data and inline it
- Size: no larger than 1em
- Color: `currentColor` (inherits text color)

### Default Behavior
Generate articles with **zero icons** in the body. Only include if template explicitly calls for a callout or byline component.

## 7. IMAGE REQUIREMENTS & PLACEMENT

### 7.1 Image sources

Images come from a three-tier fallback in `Stock_Image_Inserter::get_image_url()`:

1. **Pexels API** (preferred) — if the user has set `pexels_api_key` in settings, real keyword-relevant photos are fetched via the Pexels search API. Cached per query for 6 hours. Results are deduplicated per article via `$used_urls`.
2. **Picsum fallback** — if no Pexels key, generates seeded URLs like `https://picsum.photos/seed/{crc32}/800/450.jpg`. These are random but consistent per keyword/heading combo.

Featured images for posts use `search_pexels_image()` in `seobetter.php` (separate from the inline inserter), falling back to Picsum with `.jpg` extension to avoid the old 302-without-extension bug.

### 7.2 Placement rules (v1.5.10+)

**Inline images are inserted by `Stock_Image_Inserter::insert_images()` after article generation, before save.** Three images maximum per article. Placement rules:

- **Never place images adjacent to structural sections.** The inserter skips any H2 whose text matches `/key\s*takeaway|faq|frequently\s*asked|references|sources|bibliography|further\s*reading/i`.
- **Content-bearing H2s only.** Images land on H2 #2, #5, and #8 (not #1, not #3, not #4). If the article has fewer than 2 content H2s, zero images are inserted.
- **Below the heading, above the body.** The image markdown `![alt](url)` is written immediately after the `## Heading` line and before the section body.

**Why these rules exist:** Placing an image directly above a Key Takeaways bullet list (the old behavior) (a) looks cluttered since Key Takeaways is already visually heavy, and (b) historically triggered FM-13 — a regex bug in the link validator stripped image URLs whose hosts weren't whitelisted, leaving a stray `!` character that broke the `format_hybrid` backward walk for Key Takeaways styling detection. The placement rule is belt-and-braces alongside the regex fix in `seobetter.php:1363`.

### 7.3 Alt text generation

`Stock_Image_Inserter::generate_alt_text()` produces varied, descriptive alt text based on heading patterns:

| Heading pattern | Alt text template |
|---|---|
| Question (`what`, `how`, `why`, etc.) | `"{Question} — {keyword} visual guide"` |
| Comparison (`vs`, `versus`, `compare`) | `"{Keyword} comparison table showing features and ratings"` |
| FAQ | `"Frequently asked questions about {keyword} answered by experts"` |
| Default | Rotating templates: comparison chart / guide to / overview / breakdown / visual guide |

Alt text is clamped to 125 characters and never starts with "image of" or "picture of".

### 7.4 Rendered output

```html
<figure style="margin:1.5em 0;text-align:center">
  <img src="URL" alt="Descriptive alt text"
       width="800" height="450"
       loading="lazy" decoding="async"
       style="max-width:100%;height:auto;border-radius:8px;box-shadow:0 2px 8px rgba(0,0,0,0.1)" />
</figure>
```

In hybrid mode, images are wrapped in `<!-- wp:image -->` blocks (editable). In classic mode, the full `<figure>` HTML is returned inline.

| Attribute | Value | Purpose |
|---|---|---|
| `loading` | `lazy` | Defer off-screen images |
| `decoding` | `async` | Non-blocking decode |
| `width` / `height` | `800` / `450` (explicit) | Prevent CLS |
| `max-width` | `100%` | Responsive |
| `border-radius` | `8px` | Visual consistency |
| `box-shadow` | subtle | Lifts image from page |

### 7.5 Anti-pattern — DO NOT place images near:

| Section | Why |
|---|---|
| `## Key Takeaways` | Visual clutter + historical FM-13 trigger |
| `## Frequently Asked Questions` | Breaks Q&A rhythm |
| `## References` | References is structural, not content |
| H1 title or first paragraph | Drop cap handles hero visual; image above first content H2 competes with it |

## 9. PERFORMANCE

### Loading
- No external CSS files or @import
- No JavaScript in article output
- Lazy-loaded images (`loading="lazy"`)
- `decoding="async"` for non-blocking image decode
- `aspect-ratio` prevents cumulative layout shift

### Render
- Scoped CSS limits style recalculation to article element only
- No CSS animations by default (saves paint cycles)
- Minimal DOM nodes — semantic HTML without wrapper divs

### Indexing
- Semantic HTML (article, h1-h3, p, ul, ol, table, blockquote, figure)
- Schema JSON-LD in `<head>` via wp_head hook (separate from article HTML)
- All text content in proper HTML elements (not hidden in attributes or data-)
- Links are real `<a href>` tags crawlable by search engines and AI agents

## 10. REFERENCES SECTION (v1.5.9+)

**The AI never writes the References section.** It is built programmatically at save time by `append_references_section()` in `seobetter.php`, using metadata from the per-article Citation Pool. See `external-links-policy.md` Section 8 for the full architecture.

### How it works

1. During the "trends" step of async generation, `Citation_Pool::build()` performs a real keyword-targeted web search (DuckDuckGo / Brave / Wikipedia / article-linking Reddit+HN) and content-verifies each candidate URL. Up to 12 verified pool entries are attached to the job: `{ url, title, source_name, verified_at }`.
2. The pool is injected into every section's prompt as an `AVAILABLE CITATIONS` block. The AI may ONLY cite URLs from this list.
3. The AI is explicitly told **not** to output a References section.
4. At save time, `validate_outbound_links()` filters the body (Pass 2 pool membership + static whitelist fallback, Pass 3 RLFKV content verification).
5. `append_references_section()` walks the cleaned body, collects every pool URL the body actually cited, and appends a numbered `## References` section using **pool metadata titles** (scraped at pool-build time, never written by the AI).

### Guarantees

- **Zero fabrication surface** — every entry traces to a pool URL that was cited in the body
- **Only cited entries appear** — no orphan references
- **Titles come from the pool**, not the AI — they match the real destination page
- **If zero pool URLs were cited, no References section is appended** — the article simply ends at its last content section

### Rendered output (hybrid mode)

```markdown
## References

1. [How to Choose a Dog Bed](https://www.akc.org/expert-advice/home-living/how-to-choose-dog-bed/) — akc.org
2. [Orthopedic Pet Beds Guide](https://www.petbarn.com.au/petspot/dog/orthopedic-beds/) — petbarn.com.au
3. [Best Beds for Anxious Dogs](https://www.rspca.org.au/knowledgebase/anxious-dog-beds/) — rspca.org.au
```

In hybrid output this renders as:

```html
<!-- wp:heading -->
<h2 class="wp-block-heading">References</h2>
<!-- /wp:heading -->

<!-- wp:list {"ordered":true} -->
<ol>
  <li><a href="https://www.akc.org/expert-advice/home-living/how-to-choose-dog-bed/">How to Choose a Dog Bed</a> — akc.org</li>
  <li><a href="https://www.petbarn.com.au/petspot/dog/orthopedic-beds/">Orthopedic Pet Beds Guide</a> — petbarn.com.au</li>
  <li><a href="https://www.rspca.org.au/knowledgebase/anxious-dog-beds/">Best Beds for Anxious Dogs</a> — rspca.org.au</li>
</ol>
<!-- /wp:list -->
```

### What's prohibited

- **Fabricated URLs** — stripped by `validate_outbound_links()` before References is built
- **API endpoints** — `api.*`, `*-api.*`, `/api/`, `/v1/`, `*.herokuapp.com` all hard-fail
- **API / dataset / tool names as anchor text** — `[Dog Facts API](...)`, `[Pexels API](...)` stripped unconditionally
- **Bare homepages** — `https://www.rspca.org.au/` (no path) stripped
- **Vague anchor text** — `[here](...)`, `[this article](...)`, `[learn more](...)` stripped (no verifiable content words)
- **Misattributed citations** — links whose anchor text doesn't match the destination page content (RLFKV Pass 3, 50% content-word overlap requirement)

### When References is absent

If the research pipeline returns zero pool-relevant URLs for an obscure keyword (FM-10), OR the AI uses plain-text attributions throughout instead of hyperlinks, the article ends without a References section. This is **correct behavior** — per Joshi 2025, "an article with zero fabricated citations is a pass."

Plain-text attributions like *"According to a 2026 RSPCA report"* count equally toward GEO visibility (Gosmar & Dahl 2025 FDF metric) and are the preferred fallback.

## 11. CONTENT TYPE VARIATIONS (v1.5.4+)

Each content type affects **three layers** of the generated article:

1. **Prompt guidance** — `Async_Generator::get_prose_template()` injects type-specific writing rules into every section prompt
2. **Styled block detection** — `Content_Formatter::format_hybrid()` applies different inline styles based on heading context
3. **Schema output** — `Schema_Generator::generate()` in `includes/Schema_Generator.php` AND `build_aioseo_schema()` in `seobetter.php` emit different `@type` values and secondary schemas

### Design & schema matrix

| Content Type | Primary Schema | Secondary Schemas | Styled Blocks |
|---|---|---|---|
| `blog_post` | `BlogPosting` | FAQPage (if Q&A) | Key Takeaways, tables, callouts |
| `news_article` | `NewsArticle` | FAQPage (if Q&A) | Short paragraphs, dateline |
| `opinion` | `OpinionNewsArticle` | FAQPage | Blockquotes emphasized |
| `how_to` | `HowTo` + BreadcrumbList | FAQPage | Numbered step lists, prerequisite callouts |
| `listicle` | `Article` + `ItemList` | FAQPage | H2 per item, comparison feel |
| `review` | `Review` (with `itemReviewed` + `reviewRating`) | FAQPage | Pros/cons boxes (green/red) |
| `comparison` | `Article` + `ItemList` | FAQPage | Mandatory comparison table |
| `buying_guide` | `Article` + `Product` + `ItemList` | FAQPage | Product recommendations, pros/cons |
| `pillar_guide` | `Article` + `ItemList` (TOC) | FAQPage | Deep multi-section structure |
| `case_study` | `Article` | FAQPage | Research/results structure |
| `interview` | `Article` | FAQPage | Blockquotes for quoted responses |
| `faq_page` | `FAQPage` (primary) | — | All content as Q&A pairs |
| `recipe` | `Recipe` (with ingredients, instructions, times, yield) | — | Ingredients box (amber), numbered steps |
| `tech_article` | `TechArticle` | FAQPage | Code blocks, technical diagrams |
| `white_paper` | `Article` | — | Formal tone, no drop cap |
| `scholarly_article` | `ScholarlyArticle` | — | Citations, formal structure |
| `live_blog` | `LiveBlogPosting` | — | Reverse-chronological updates |
| `press_release` | `NewsArticle` | — | Dateline, formal announcement |
| `personal_essay` | `BlogPosting` | — | First person, longer paragraphs |
| `glossary_definition` | `DefinedTerm` + `DefinedTermSet` | — | Short, definition-first |
| `sponsored` | `BlogPosting` (AdvertiserContent) | — | Disclosure block |

### Design adjustments by type

| Content Type | Design Adjustments |
|---|---|
| Recipe | Ingredients box (amber), numbered step styling, prep/cook time metadata, recipe card visual |
| Review | Pros/cons boxes (green/red), verdict section, rating display |
| Comparison | Mandatory comparison table, criterion winner indicators |
| News Article | Short paragraphs, no drop cap, dateline styling |
| FAQ Page | Q&A card styling for each pair |
| How-To | Numbered step cards, prerequisite box, HowTo schema with `totalTime` |
| Glossary | Short (800 words), definition-first paragraph, no FAQ section |
| Listicle | H2 per item (detected and promoted to ItemList schema) |
| Buying Guide | Product comparison table, Product schema with aggregateRating |

### Schema variation — non-obvious rules

- **FAQ section is auto-detected** by `Schema_Generator::generate_faq_schema()` — any H2/H3 matching `/^(what|how|why|when|where|who|which|can|does|is|are|should|do)|\?$/i` with a followup paragraph of ≥10 chars becomes a `Question` in `FAQPage` schema
- **HowTo schema is auto-generated** when the title contains "how to" AND the body has at least 2 ordered-list items, regardless of the declared content type
- **ItemList schema** applies to `listicle`, `buying_guide`, `comparison`, `pillar_guide` — extracts H2 headings (excluding structural sections)
- **DefinedTerm** replaces the base Article schema entirely for `glossary_definition`, not added alongside
- **BreadcrumbList** is always added as a secondary schema regardless of type

See `includes/Schema_Generator.php` and `seobetter.php::build_aioseo_schema()` for implementation.

## 12. DESIGN CHECKLIST

Every generated article MUST have:

**Structure & format**
- [ ] Hybrid format output (`format_hybrid()`) — native Gutenberg blocks for editable content, `wp:html` for styled elements
- [ ] `<h1>` with article title (not title-case-corrupted)
- [ ] `*Last Updated: Month Year*` small italic paragraph immediately after H1
- [ ] `## Key Takeaways` section with 3–5 bullets, rendered as styled `wp:html` block (purple left-border gradient)
- [ ] `## Frequently Asked Questions` section with 5 Q&A pairs (H3 questions + paragraph answers)
- [ ] Semantic HTML throughout (article, headings, p, ul, ol, table, figure)

**Typography & styling**
- [ ] System font stacks (no external fonts)
- [ ] Fluid heading sizes via `clamp()` (classic mode preview only)
- [ ] Accent color applied: H2 text, borders, links, table headers (user's color picker value, default `#764ba2`)
- [ ] No emoji in article body
- [ ] No AI tells (delve, tapestry, pivotal, cornerstone, etc.) — see `Async_Generator.php::get_system_prompt()` banned words list

**Styled blocks (wp:html)**
- [ ] Key Takeaways box if article has a Key Takeaways section (detected by `/key\s*takeaway/i` on preceding heading)
- [ ] Comparison tables rendered with styled header (accent bg, white text)
- [ ] Pros/cons boxes if a section has pros/cons lists (green/red bg detection)
- [ ] Callout boxes for paragraphs starting with `Tip:`, `Note:`, `Warning:` (blue/amber/red)
- [ ] Blockquotes with accent left-border and italic styling

**Images (v1.5.10+)**
- [ ] Up to 3 images placed on content-bearing H2s (#2, #5, #8) via `Stock_Image_Inserter`
- [ ] **Never** placed adjacent to Key Takeaways, FAQ, References, or similar structural sections
- [ ] Lazy-loaded with explicit width/height (prevent CLS)
- [ ] Pexels (if API key) or Picsum fallback
- [ ] Descriptive alt text, never "image of" or "picture of"

**Citations & References (v1.5.9+)**
- [ ] All hyperlinks match a URL in the per-article Citation Pool OR the static trusted whitelist
- [ ] Hyperlinks must pass RLFKV Pass 3 content verification (anchor text words ≥50% match destination)
- [ ] References section **auto-appended by plugin** — never written by AI
- [ ] References section only present if body cited at least one pool URL
- [ ] Zero hallucinated URLs, zero API endpoints, zero bare homepages, zero vague anchor text
- [ ] Plain-text attributions ("According to a 2026 RSPCA report") for claims not in the pool

**Technical**
