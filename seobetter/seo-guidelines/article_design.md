## 1. DESIGN PHILOSOPHY

Articles must look like they came from a design-conscious publication (The Verge, Stripe Press, NYT Magazine) ŌĆö not a Notion page full of emoji headers or a chatbot response.

### Core Principles
1. **Hybrid format** ŌĆö Native Gutenberg blocks for editable content + `wp:html` blocks with inline styles for styled elements only.
2. **Editable** ŌĆö Headings, paragraphs, standard lists, images are native WordPress blocks. Users can edit them in the block editor.
3. **Styled where needed** ŌĆö Key takeaways, pros/cons, tables, callouts, blockquotes use `wp:html` with inline styles and `!important` to override theme CSS.
4. **Schema injected** ŌĆö JSON-LD schema is appended as a `wp:html` block at the end of every article's `post_content`.
5. **Responsive** ŌĆö Responsive images with `loading="lazy"`, mobile-friendly tables with `overflow-x:auto`.
6. **Accessible** ŌĆö Semantic HTML, sufficient contrast ratios, no color-only indicators.
7. **Theme-compatible** ŌĆö Uses `!important` on styled elements to override any WordPress theme CSS. Editable blocks inherit theme fonts and colors.

```html

  <h1>Article Title</h1>

  </div>

  <h2>Section Heading</h2>
  <p>Content paragraph...</p>
```

### Rules
- Output a single `<article>` element with class `sb-[6-char-uid]`
- Include one `<style>` tag at the top with every selector prefixed by the scoping class
- Do NOT use Tailwind, Bootstrap, or any framework classes
- Do NOT use `<link>` tags, `@import`, or `<script>` tags
- Return only the HTML block ŌĆö no markdown fences, no DOCTYPE

### 5.2 Pros Box (`.sb-pros`)

### 5.3 Cons Box (`.sb-cons`)
- Background: light red (`#fef2f2`)
- Border: 1px solid red (`#fecaca`)
- Used when preceding heading contains "cons", "disadvantages", "drawbacks"

### 5.4 Ingredients Box (`.sb-ingredients`)
- Background: light amber (`#fffbeb`)
- Border: 1px solid amber (`#fde68a`)
- Used when preceding heading contains "ingredients", "supplies", "what you need"

### 5.5 Callout Boxes (`.sb-callout`)
Detected from paragraph text starting with "Tip:", "Note:", or "Warning:"

| Type | Class | Background | Left Border |
|---|---|---|---|
| Tip | `.sb-callout-tip` | `#eff6ff` (light blue) | `#3b82f6` (blue) |
| Note | `.sb-callout-note` | `#fffbeb` (light amber) | `#f59e0b` (amber) |
| Warning | `.sb-callout-warn` | `#fef2f2` (light red) | `#ef4444` (red) |

### 5.6 Blockquotes
- Left border: 4px solid accent
- Background: surface color
- Border radius: 0 8px 8px 0
- Font style: italic
- Used for expert quotes and pull quotes

### 5.7 Tables
- Full width, border-collapse
- Header: accent background, white text, small caps
- Even rows: surface background (zebra striping)
- Cells: comfortable padding (0.75em 1em)
- Wrapper: `overflow-x: auto` for mobile horizontal scroll
- Subtle box shadow

### 5.8 Code and Pre Blocks
- Inline code: light gray background, small padding, rounded corners
- Code blocks: dark background (#1f2937), light text, rounded corners, horizontal scroll

### 5.9 Stat Callout (`v1.5.14+`)
Auto-detected from any paragraph containing a prominent statistic ŌĆö `\d%`, `X out of Y`, `X in N`.
- Layout: flex row with the extracted stat as a 2em bold number on the left, body text on the right
- Background: light purple (`#faf5ff`), accent left border (4px)
- Trigger regex: `(\d{1,3}(?:[.,]\d+)?\s*%)` OR `\b\d{1,3}\s+(?:out\s+of|in)\s+\d{1,4}\b`
- Source: `Content_Formatter.php::format_hybrid()` paragraph branch ŌĆö Stat callout

### 5.10 Expert Quote (`v1.5.14+`)
Auto-detected from any paragraph matching `"Quote text" ŌĆö Name, Title` (em/en/regular dash, straight or curly quotes).
- Layout: italic blockquote with the quote in 1.1em, attribution in a smaller `<footer>` line below
- Background: surface gray (`#f9fafb`), accent left border (4px), border-radius 0 8px 8px 0
- Trigger regex (Unicode-aware): `^["\x{201C}]([^"\x{201D}]{20,})["\x{201D}]\s*[\x{2014}\x{2013}\-]\s*([A-Z][a-zA-Z\s.\']+?)(?:,\s*(.+?))?[.\s]*$`
- Source: `Content_Formatter.php::format_hybrid()` paragraph branch ŌĆö Expert quote

### 5.11 Definition Box (`v1.5.14+`)
Auto-detected from paragraphs starting with `**Term**:` (i.e. raw markdown `**Term**: explanation`).
- Layout: term in accent color + semibold, separator middot, explanation in body color
- Background: light gray (`#f8fafc`), 1px border (`#e2e8f0`), border-radius 8px
- Trigger regex: `^<strong>([^<]{2,40})</strong>\s*[:ŌĆö-]\s*(.+)$`
- Source: `Content_Formatter.php::format_hybrid()` paragraph branch ŌĆö Definition box

### 5.12 Did-You-Know Box (`v1.5.14+`)
Auto-detected from paragraphs starting with `Did you know` or `Fun fact`.
- Layout: small uppercase eyebrow label "DID YOU KNOW?" in amber, body text in dark amber underneath
- Background: soft yellow (`#fefce8`), amber-yellow left border (`#eab308`)
- **No icon, no emoji** ŌĆö typography only per ┬¦6
- Trigger regex (case-insensitive): `^(did\s*you\s*know|fun\s*fact)\??\s*[:ŌĆö-]?\s*(.*)$`
- Cap: max 1 per article (the AI prompt also enforces this)
- Source: `Content_Formatter.php::format_hybrid()` paragraph branch ŌĆö Did You Know box

### 5.13 Highlight Sentence (`v1.5.14+`)
Auto-detected when an entire paragraph is a single bold sentence (raw markdown `**The whole sentence is bold.**`).
- Layout: large 1.15em font, accent text color, 6px accent left border
- Background: light surface purple (`#faf5ff`), border-radius 0 8px 8px 0
- Used by the AI to mark the single most important takeaway sentence in a section (max 2 per article ŌĆö prompt-enforced)
- Trigger regex: `^<strong>([^<].*?)</strong>[\s.!?]*$` plus structural check that inner content has no nested HTML
- Source: `Content_Formatter.php::format_hybrid()` paragraph branch ŌĆö Highlight sentence

### 5.14 HowTo Step Boxes (`v1.5.14+`)
Auto-detected when `content_type === 'how_to'` AND the list is ordered (`<ol>`) AND it's not already classified as Pros/Cons/Ingredients/Takeaways. Each `<li>` becomes its own row.
- Layout: flex row per step ŌĆö circular numbered badge (36px) on the left in accent color with white number, step text on the right
- Step card: light surface background (`#f8fafc`), 1px border (`#e2e8f0`), border-radius 10px, 1em gap between steps
- **No SVG icons, no checkmarks** ŌĆö the number itself is the visual marker
- Threading: `format_hybrid()` reads `$options['content_type']` (passed in by `seobetter.php::rest_save_draft()` and `Async_Generator::assemble_final()`)
- Source: `Content_Formatter.php::format_hybrid()` list branch ŌĆö HowTo step boxes

### 5.15 Widened triggers for existing boxes (`v1.5.14+`)
Existing boxes 5.1ŌĆō5.5 now match more synonyms in the preceding H2:
- **Takeaways**: also `key insight`, `main point`, `at a glance`, `tldr`, `tl;dr`, `what to know`, `the bottom line`
- **Pros**: also `upside`, `highlight`
- **Cons**: also `downside`, `limitation`, `trade-off`
- **Ingredients**: also `materials`, `tools`, `prerequisites`

This roughly doubles the existing styled-block hit rate without changing the prompt.

## 6. ICON RULES (STRICT)

### Inside Generated Articles
- **NEVER** place icons next to headings, list items, or inline with sentences
- **NEVER** use emoji as icons in body copy
- No checkmark bullets, no decorative icons beside H2s, no emoji headers
- This is the single clearest tell that content was AI-generated

### Icons Permitted ONLY In
1. Small SVG glyphs inside callout box corners (info icon, sized 1em, `currentColor`)
2. A single icon in the Key Takeaways box header (not per item)
3. Social share or author byline sections

### When Icons Are Used
- Embed as inline SVG directly in the HTML
- Do NOT reference Lucide or Iconify as runtime dependencies
- Copy raw SVG path data and inline it
- Size: no larger than 1em
- Color: `currentColor` (inherits text color)

### Default Behavior
Generate articles with **zero icons** in the body. Only include if template explicitly calls for a callout or byline component.

## 7. IMAGE REQUIREMENTS & PLACEMENT

### 7.1 Image sources

Images come from a three-tier fallback in `Stock_Image_Inserter::get_image_url()`:

1. **Pexels API** (preferred) ŌĆö if the user has set `pexels_api_key` in settings, real keyword-relevant photos are fetched via the Pexels search API. Cached per query for 6 hours. Results are deduplicated per article via `$used_urls`.
2. **Picsum fallback** ŌĆö if no Pexels key, generates seeded URLs like `https://picsum.photos/seed/{crc32}/800/450.jpg`. These are random but consistent per keyword/heading combo.

Featured images for posts use `search_pexels_image()` in `seobetter.php` (separate from the inline inserter), falling back to Picsum with `.jpg` extension to avoid the old 302-without-extension bug.

### 7.2 Placement rules (v1.5.10+)

**Inline images are inserted by `Stock_Image_Inserter::insert_images()` after article generation, before save.** Three images maximum per article. Placement rules:

- **Never place images adjacent to structural sections.** The inserter skips any H2 whose text matches `/key\s*takeaway|faq|frequently\s*asked|references|sources|bibliography|further\s*reading/i`.
- **Content-bearing H2s only.** Images land on H2 #2, #5, and #8 (not #1, not #3, not #4). If the article has fewer than 2 content H2s, zero images are inserted.
- **Below the heading, above the body.** The image markdown `![alt](url)` is written immediately after the `## Heading` line and before the section body.

**Why these rules exist:** Placing an image directly above a Key Takeaways bullet list (the old behavior) (a) looks cluttered since Key Takeaways is already visually heavy, and (b) historically triggered FM-13 ŌĆö a regex bug in the link validator stripped image URLs whose hosts weren't whitelisted, leaving a stray `!` character that broke the `format_hybrid` backward walk for Key Takeaways styling detection. The placement rule is belt-and-braces alongside the regex fix in `seobetter.php:1363`.

### 7.3 Alt text generation

`Stock_Image_Inserter::generate_alt_text()` produces varied, descriptive alt text based on heading patterns:

| Heading pattern | Alt text template |
|---|---|
| Question (`what`, `how`, `why`, etc.) | `"{Question} ŌĆö {keyword} visual guide"` |
| Comparison (`vs`, `versus`, `compare`) | `"{Keyword} comparison table showing features and ratings"` |
| FAQ | `"Frequently asked questions about {keyword} answered by experts"` |
| Default | Rotating templates: comparison chart / guide to / overview / breakdown / visual guide |

Alt text is clamped to 125 characters and never starts with "image of" or "picture of".

### 7.4 Rendered output

```html
<figure style="margin:1.5em 0;text-align:center">
  <img src="URL" alt="Descriptive alt text"
       width="800" height="450"
       loading="lazy" decoding="async"
       style="max-width:100%;height:auto;border-radius:8px;box-shadow:0 2px 8px rgba(0,0,0,0.1)" />
</figure>
```

In hybrid mode, images are wrapped in `<!-- wp:image -->` blocks (editable). In classic mode, the full `<figure>` HTML is returned inline.

| Attribute | Value | Purpose |
|---|---|---|
| `loading` | `lazy` | Defer off-screen images |
| `decoding` | `async` | Non-blocking decode |
| `width` / `height` | `800` / `450` (explicit) | Prevent CLS |
| `max-width` | `100%` | Responsive |
| `border-radius` | `8px` | Visual consistency |
| `box-shadow` | subtle | Lifts image from page |

### 7.5 Anti-pattern ŌĆö DO NOT place images near:

| Section | Why |
|---|---|
| `## Key Takeaways` | Visual clutter + historical FM-13 trigger |
| `## Frequently Asked Questions` | Breaks Q&A rhythm |
| `## References` | References is structural, not content |
| H1 title or first paragraph | Drop cap handles hero visual; image above first content H2 competes with it |

## 9. PERFORMANCE

### Loading
- No external CSS files or @import
- No JavaScript in article output
- Lazy-loaded images (`loading="lazy"`)
- `decoding="async"` for non-blocking image decode
- `aspect-ratio` prevents cumulative layout shift

### Render
- Scoped CSS limits style recalculation to article element only
- No CSS animations by default (saves paint cycles)
- Minimal DOM nodes ŌĆö semantic HTML without wrapper divs

### Indexing
- Semantic HTML (article, h1-h3, p, ul, ol, table, blockquote, figure)
- Schema JSON-LD in `<head>` via wp_head hook (separate from article HTML)
- All text content in proper HTML elements (not hidden in attributes or data-)
- Links are real `<a href>` tags crawlable by search engines and AI agents

## 10. REFERENCES SECTION (v1.5.9+)

**The AI never writes the References section.** It is built programmatically at save time by `append_references_section()` in `seobetter.php`, using metadata from the per-article Citation Pool. See `external-links-policy.md` Section 8 for the full architecture.

### How it works

1. During the "trends" step of async generation, `Citation_Pool::build()` performs a real keyword-targeted web search (DuckDuckGo / Brave / Wikipedia / article-linking Reddit+HN) and content-verifies each candidate URL. Up to 12 verified pool entries are attached to the job: `{ url, title, source_name, verified_at }`.
2. The pool is injected into every section's prompt as an `AVAILABLE CITATIONS` block. The AI may ONLY cite URLs from this list.
3. The AI is explicitly told **not** to output a References section.
4. At save time, `validate_outbound_links()` filters the body (Pass 2 pool membership + static whitelist fallback, Pass 3 RLFKV content verification).
5. `append_references_section()` walks the cleaned body, collects every pool URL the body actually cited, and appends a numbered `## References` section using **pool metadata titles** (scraped at pool-build time, never written by the AI).

### Guarantees

- **Zero fabrication surface** ŌĆö every entry traces to a pool URL that was cited in the body
- **Only cited entries appear** ŌĆö no orphan references
- **Titles come from the pool**, not the AI ŌĆö they match the real destination page
- **If zero pool URLs were cited, no References section is appended** ŌĆö the article simply ends at its last content section

### Rendered output (hybrid mode)

```markdown
## References

1. [How to Choose a Dog Bed](https://www.akc.org/expert-advice/home-living/how-to-choose-dog-bed/) ŌĆö akc.org
2. [Orthopedic Pet Beds Guide](https://www.petbarn.com.au/petspot/dog/orthopedic-beds/) ŌĆö petbarn.com.au
3. [Best Beds for Anxious Dogs](https://www.rspca.org.au/knowledgebase/anxious-dog-beds/) ŌĆö rspca.org.au
```

In hybrid output this renders as:

```html
<!-- wp:heading -->
<h2 class="wp-block-heading">References</h2>
<!-- /wp:heading -->

<!-- wp:list {"ordered":true} -->
<ol>
  <li><a href="https://www.akc.org/expert-advice/home-living/how-to-choose-dog-bed/">How to Choose a Dog Bed</a> ŌĆö akc.org</li>
  <li><a href="https://www.petbarn.com.au/petspot/dog/orthopedic-beds/">Orthopedic Pet Beds Guide</a> ŌĆö petbarn.com.au</li>
  <li><a href="https://www.rspca.org.au/knowledgebase/anxious-dog-beds/">Best Beds for Anxious Dogs</a> ŌĆö rspca.org.au</li>
</ol>
<!-- /wp:list -->
```

### What's prohibited

- **Fabricated URLs** ŌĆö stripped by `validate_outbound_links()` before References is built
- **API endpoints** ŌĆö `api.*`, `*-api.*`, `/api/`, `/v1/`, `*.herokuapp.com` all hard-fail
- **API / dataset / tool names as anchor text** ŌĆö `[Dog Facts API](...)`, `[Pexels API](...)` stripped unconditionally
- **Bare homepages** ŌĆö `https://www.rspca.org.au/` (no path) stripped
- **Vague anchor text** ŌĆö `[here](...)`, `[this article](...)`, `[learn more](...)` stripped (no verifiable content words)
- **Misattributed citations** ŌĆö links whose anchor text doesn't match the destination page content (RLFKV Pass 3, 50% content-word overlap requirement)

### When References is absent

If the research pipeline returns zero pool-relevant URLs for an obscure keyword (FM-10), OR the AI uses plain-text attributions throughout instead of hyperlinks, the article ends without a References section. This is **correct behavior** ŌĆö per Joshi 2025, "an article with zero fabricated citations is a pass."

Plain-text attributions like *"According to a 2026 RSPCA report"* count equally toward GEO visibility (Gosmar & Dahl 2025 FDF metric) and are the preferred fallback.

## 11. CONTENT TYPE VARIATIONS (v1.5.4+)

Each content type affects **three layers** of the generated article:

1. **Prompt guidance** ŌĆö `Async_Generator::get_prose_template()` injects type-specific writing rules into every section prompt
2. **Styled block detection** ŌĆö `Content_Formatter::format_hybrid()` applies different inline styles based on heading context
3. **Schema output** ŌĆö `Schema_Generator::generate()` in `includes/Schema_Generator.php` AND `build_aioseo_schema()` in `seobetter.php` emit different `@type` values and secondary schemas

### Design & schema matrix

| Content Type | Primary Schema | Secondary Schemas | Styled Blocks |
|---|---|---|---|
| `blog_post` | `BlogPosting` | FAQPage (if Q&A) | Key Takeaways, tables, callouts |
| `news_article` | `NewsArticle` | FAQPage (if Q&A) | Short paragraphs, dateline |
| `opinion` | `OpinionNewsArticle` | FAQPage | Blockquotes emphasized |
| `how_to` | `HowTo` + BreadcrumbList | FAQPage | Numbered step lists, prerequisite callouts |
| `listicle` | `Article` + `ItemList` | FAQPage | H2 per item, comparison feel |
| `review` | `Review` (with `itemReviewed` + `reviewRating`) | FAQPage | Pros/cons boxes (green/red) |

### Schema variation ŌĆö non-obvious rules

- **FAQ section is auto-detected** by `Schema_Generator::generate_faq_schema()` ŌĆö any H2/H3 matching `/^(what|how|why|when|where|who|which|can|does|is|are|should|do)|\?$/i` with a followup paragraph of Ōēź10 chars becomes a `Question` in `FAQPage` schema
- **HowTo schema is auto-generated** when the title contains "how to" AND the body has at least 2 ordered-list items, regardless of the declared content type
- **ItemList schema** applies to `listicle`, `buying_guide`, `comparison`, `pillar_guide` ŌĆö extracts H2 headings (excluding structural sections)
- **DefinedTerm** replaces the base Article schema entirely for `glossary_definition`, not added alongside
- **BreadcrumbList** is always added as a secondary schema regardless of type

See `includes/Schema_Generator.php` and `seobetter.php::build_aioseo_schema()` for implementation.
